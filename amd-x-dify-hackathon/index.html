Track 1 构建模式

好用的多模态产品，
关键在于还能继续改。

别把第一次生成当成终点，把检查、调整和再次生成都放进 Workflow。

Build loop
01

先问清楚

Prompt、参考素材、目标格式、风格和限制条件

02

整理素材

检查文件、统一参数，再组装生成请求

03

在 Radeon 上生成

调用 ROCm 后端，同时记录运行数据

04

检查质量

看结果是否成功、清晰、一致，也符合用户预期

05

交付或重试

结果满意就交付；不满意就改参数再跑一次

Demo 最好既跑一遍成功流程，也现场改一次。
调一次模型只是功能，能持续调整才像产品。先问清用户要什么，整理好素材，在 Radeon 上生成，再检查结果；满意就交付，不满意就回到流程里继续改。现场演示一次修改，比单纯展示成品更能说明 Workflow 的价值。

评委想看的证据

别只报一个速度数字，
让数据和结果互相印证。

数字要有上下文

延迟

从发出请求，到看到首个结果和完整结果，各用了多久

GPU 显存

在明确的模型、分辨率和 Batch 设置下记录峰值 VRAM

稳定性

连续跑几次成功了多少次，失败后能不能顺利恢复

输出质量

结果是否清晰、稳定，和输入素材、用户要求是否一致

多样性

能不能生成不同但都可用的结果，又不跑偏主题和风格

放在一起：运行环境 + 参数 + 测量值 + 输出结果数字离开上下文，就没有说服力。
不需要堆很多 Benchmark，挑一组小而可信的数据就够了：延迟、峰值显存、连续运行是否稳定、结果质量和多样性。每个数字旁边都写清环境和参数，评委才知道这个结果是怎么来的，也更容易复现。
