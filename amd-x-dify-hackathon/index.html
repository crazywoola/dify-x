一个系统，两类职责

用 Dify 编排，
用 Radeon 生成和加速。

Dify × AMD
控制平面

Dify Workflow

  • 接收文本、图像、音频、视频和参数
  • 完成分支、迭代、校验和失败恢复
  • 通过模型、工具或 HTTP 节点连接外部服务
  • 直接提供 UI 与 API，无需重写编排代码
计算平面

AMD Radeon + ROCm

  • 执行生成或媒体转换工作负载
  • 让执行路径在 Demo 中清晰可见
  • 测量延迟、显存、稳定性与吞吐
  • 把媒体资产和元数据返回 Workflow
产品边界就在交接点：Dify 发送定义清晰的生成任务，Radeon 返回可测量的媒体资产。
把职责保持简单。Dify 是控制平面：输入、分支、循环、工具、UI 和 API。Radeon 是计算平面：运行成本高的生成或转换任务，并返回资产和证据。这样分层既便于调试，也能让 AMD 的贡献在演示中一眼可见。
参考架构

让每一次交接都清晰可见。

Workflow 负责状态与决策，Radeon 服务负责 GPU 工作负载。

产品输出

连接方式：Tool 节点 · HTTP Request · 支持时使用 OpenAI-compatible API
这是一套可以复用的架构。输入进入 Dify，由 Workflow 标准化并决定下一步。工具或 HTTP 请求把边界清晰的任务发送给 Radeon 后端。结果以文件或 URL 加元数据的形式返回。不要隐藏这条边界，请把请求、执行和返回资产都展示出来。

Track 1 构建模式

设计的是循环，
不只是第一次生成。

优秀的创作体验会把质量检查与重新生成放进 Workflow。

Build loop
01

收集意图

Prompt、参考素材、目标格式、风格与约束

02

准备素材

校验文件、标准化元数据并构造生成请求

03

在 Radeon 上生成

调用 ROCm 工作负载并记录执行证据

04

检查质量

检查成功状态、尺寸、安全、一致性和用户标准

05

交付或重试

返回资产，或修改参数后再次进入循环

好的 Demo 会同时展示一次成功路径和一次受控迭代。
一次生成调用只是功能，循环才是产品。收集足够的意图、准备请求、在 Radeon 上执行、检查结果，然后交付或重试。演示时最好同时展示成功路径和一次有目的的修改。这能证明 Workflow 提供的不只是一个模型 Playground。

拒绝虚荣指标

延迟

从 Workflow 发出请求到首个与最终可用结果的时间

GPU 显存

明确模型、分辨率和 Batch 设置下的峰值 VRAM

稳定性

多次运行成功率，以及失败时是否能干净恢复

输出质量

分辨率、保真度、一致性与场景特定标准

多样性

在保持 Prompt 与风格一致的同时产生多种可用结果

一起记录：环境 + 参数 + 测量值 + 输出结果永远不要脱离上下文展示一个数字。
我们不会虚构 Benchmark 数字，而是记录一组规模小但可信的证据：延迟、峰值显存、重复运行稳定性、输出质量和有效多样性。每个数字旁边都要标注环境和参数。这样，运行表现才是评委能够信任和复现的证据。
