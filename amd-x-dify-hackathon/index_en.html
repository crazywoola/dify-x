Dify × AMD
CONTROL PLANE

Dify Workflow

  • Collect text, images, audio, video, and parameters
  • Branch, iterate, validate, and recover from failures
  • Connect models and services through model, tool, or HTTP nodes
  • Expose a UI and API without rebuilding orchestration code
COMPUTE PLANE

AMD Radeon + ROCm

  • Run the generation or transformation workload
  • Keep the execution path visible in the demo
  • Measure latency, memory use, stability, and throughput
  • Return media assets and metadata to the workflow
The product boundary is the handoff: Dify sends a well-defined generation job; Radeon returns a measurable asset.
Keep the responsibilities simple. Dify is the control plane: inputs, branching, loops, tools, UI, and API. Radeon is the compute plane: it runs the expensive generation or transformation and returns an asset plus evidence. This separation makes the workflow easier to debug and the AMD contribution easier to demonstrate.
Reference architecture

Make every handoff explicit.

The workflow owns state and decisions. The Radeon service owns the GPU workload.

End to end

Multimodal input

Prompt, references, style, format, and constraints

→

Dify Workflow

Prepare inputs · route · call tools · inspect · retry

→

Radeon generation service

Model inference or media processing accelerated with ROCm

→

Product output

Asset, metadata, preview, download, and run evidence

Transport: Tool node · HTTP Request · OpenAI-compatible API when supported
Here is the reusable architecture. Inputs enter Dify, where we normalize them and decide what should happen. A tool or HTTP request sends a bounded job to the Radeon-backed service. The result comes back as a file or URL plus metadata. Do not hide that boundary—the request, execution, and returned asset are the proof that the system works end to end.

Track 1 build pattern

Design the loop, not just the first generation.

The winning experience makes quality control and iteration part of the workflow.

In the demo, show both a happy path and one purposeful revision. That single iteration proves the workflow adds value beyond a model playground.

Judge-ready evidence

Measure what users feel
and what the GPU proves.

No vanity metrics

Latency

Time from workflow request to usable first and final output

GPU memory

Peak VRAM under the exact model, resolution, and batch settings

Stability

Success rate across repeated runs and clean failure handling

Output quality

Resolution, fidelity, consistency, and scenario-specific criteria

Diversity

Multiple useful outputs without losing prompt or style alignment

Record together: environment + settings + measurement + outputNever present a number without its context.
We will not invent benchmark numbers. Instead, capture a small, credible evidence set: latency, peak GPU memory, stability over repeated runs, output quality, and useful diversity. Always record the environment and settings next to the number. This turns operational performance into something the judges can trust and reproduce.
