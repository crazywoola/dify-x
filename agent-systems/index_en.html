Part 1

The Evolution of AI Systems

Moving beyond one-shot prompts into fully orchestrated, controllable agent architectures.

Transition: "Let's start with why we're here — and what changed."

Three Generations of LLM Applications

Each generation unlocks new value — and new complexity.

Gen 1

Prompt → Response

Single-turn completions. No memory, no tools, no state.

ChatGPT wrappers one-shot summarizers
Gen 2

Pipeline Orchestration

Chained nodes with data transformation, RAG, and conditional branching.

LangChain Dify Workflow
Gen 3 · Now

Agent Systems

Pause, resume, branch on human input. Reusable skills. Sandboxed execution. Collaborative authoring.

HITL Skills Sandbox
Gen 1 → Gen 2 transition happened fast (2022-2023). Gen 3 is where production teams are hitting friction today — compliance, repeatability, team scale. That's what Dify's latest releases directly address.

Three Architectural Advances

The features that define production-grade agent systems in Dify's latest releases.

Human-in-the-Loop Nodes
Dedicated execution nodes that pause a workflow and wait for human approval, override, or rejection before continuing. Enables auditable, compliance-friendly AI.
Agent × Skills
Encapsulated, reusable skill modules that decouple reasoning from execution. Agents compose skills dynamically, reducing duplication and enabling scalable multi-step designs.
Sandboxed Runtime & Collaboration
Isolated code execution environment for Python / JS nodes, plus team-level workflow co-authoring with role-based editing and version history.
Each feature maps to a real production pain-point: HITL → compliance & audit trails Skills → code duplication & maintenance burden Sandbox + Collab → team scale & security
Part 2

Human-in-the-Loop (HITL)

Controllable, auditable AI — workflows that pause for humans and resume on decision.

Available in Workflow and Chatflow modes
Key message: "AI can be wrong. HITL is the safety net that makes it deployable in regulated environments."

The Problem with Fully Autonomous AI

Real compliance scenarios need human checkpoints — not trust-me-AI.

⚠️

No Audit Trail

Fully autonomous pipelines produce outputs with no record of who decided what, making compliance reporting impossible.

🔁

Silent Errors Cascade

A bad intermediate output propagates downstream before anyone notices — sometimes triggering irreversible side effects.

🔒

Regulatory Blockers

Finance, healthcare, and legal AI deployments require documented human sign-off at key decision points.

✅

HITL Nodes solve this

Insert a pause gate anywhere in the workflow. A human reviews context, approves or overrides, and execution resumes with full trace.

Concrete industries: medical report generation, contract drafting, financial anomaly review, moderation queues.

How HITL Nodes Work

1
Trigger
Workflow runs to the HITL node boundary
2
Pause
Execution suspends; context + LLM output surfaced to reviewer
3
Decide
Human approves, edits, or rejects with optional comment
4
Resume
Workflow continues with the human decision recorded in trace
Decision Outcomes
  • Approve — continue with AI output as-is
  • Edit & Approve — override content, then continue
  • Reject — branch to fallback or terminate
Audit Capture
  • Reviewer identity & timestamp
  • Original AI output vs. final value
  • Decision comment / rationale
Demo cue: show the HITL node in Dify canvas → trigger a run → open the approval inbox → approve → show trace. Emphasize: the workflow is truly paused — it does not time-out by default (configurable).

HITL in Practice

Real deployment patterns across compliance-sensitive industries.

Contract Review

LLM drafts clause summaries; legal team approves before client delivery.

Medical Triage

AI suggests urgency score; clinician confirms before routing patient.

Financial Alerts

Anomaly detection flags a transaction; analyst approves escalation or dismisses.

Part 3

Agent × Skills

Modular, reusable skill-based agent design — decouple reasoning from execution at scale.

Works across Workflow and Agent modes
Transition: "Great — humans are in the loop. Now, how do we make the AI side more maintainable and scalable?"

The Monolithic Agent Problem

As agents grow, they become tangled, hard to test, and impossible to reuse.

Before — Monolithic Agent
Agent Prompt (500+ tokens)
search logic format rules API calls error handling output schema retry logic
Problems
duplicated across agents hard to test in isolation version drift
After — Skill-Based Agent
Agent (thin orchestrator)
goal context skill calls
Skill Library (shared)
search_web format_report call_api retry_safe
Analogy: Skills are like functions in software — you don't inline the sort algorithm every time. Same principle applies to LLM reasoning building blocks.

What Is a Skill?

A versioned, testable, composable execution unit with defined inputs and outputs.

Encapsulated

Self-contained logic with explicit input schema, output schema, and internal prompt.

Reusable

Published once, invoked from any Agent or Workflow across your workspace.

Testable

Run skills in isolation with fixture inputs — no need to trigger the full agent flow.

Versioned

Publish major/minor versions; agents pin to a version to avoid breakage on update.

Dynamic Variables

Skill inputs are assembled at runtime from conversation context, prior node outputs, or static config.

Composable

Chain multiple skills in sequence or fan-out in parallel inside a single agent step.

Demo cue: Open Skill Editor — show schema definition → publish → add to an agent → run. Highlight the variable assembly panel where you map context → skill inputs.

Skill-Based Agent Architecture

User / Trigger
chat message API call scheduled event webhook
Agent (Reasoning Layer)
goal decomposition skill selection variable assembly result synthesis
Skill Library (Execution Layer)
web_search v2 summarize_doc v1 send_email v3 query_db v1 validate_output v2 format_json v1
External Systems
APIs databases knowledge bases plugins HITL gates
Skills live in the Execution Layer — agents in the Reasoning Layer. Separating these concerns is what makes large agent systems maintainable.
This layered view maps directly to how software architects think about service layers. Reasoning: "what to do?" — Execution: "how to do it?" The Skills model makes both layers independently testable and scalable.
Part 4

Sandboxed Runtime & Collaboration

Secure code execution + team-scale workflow authoring — the production infrastructure layer.

Available in self-hosted and Dify Cloud
Transition: "The last piece: running code safely and letting teams work together."

Sandboxed Code Execution

Python and JavaScript nodes run in an isolated environment — no shared state, no host access.

Isolation Guarantees
  • No access to host filesystem
  • Network calls restricted by allowlist
  • Memory & CPU limits enforced per run
  • Execution timeout configurable per node
Supported Runtimes
Python 3.11+ JavaScript (Node 20) Custom packages via pip/npm
code_node.py 
# Runs in isolated sandbox — no host access
def main(inputs: dict) -> dict:
    text = inputs["content"]
    words = len(text.split())
    # Custom transform logic
    summary = text[:200] + "..."
    return {
        "summary": summary,
        "word_count": words
    }
Self-hosted deployments: enable the SANDBOX service in docker-compose to activate isolation. Cloud edition is sandboxed by default.
Key message: "You can now write arbitrary transformation code in your workflows without opening a security hole." Demo cue: add a Code node, paste a Python function, run — show the output in the trace.

Collaborative Workflow Development

Multiple team members author, review, and ship workflows together — with roles and version history.

Role-Based Access

Viewer, Editor, and Owner roles per workspace. Non-owners can draft but not publish.

Version History

Every publish creates a named snapshot. Roll back in one click if a new version degrades performance.

Draft → Review → Publish

Git-like lifecycle: draft changes, request peer review, merge to live — all inside Dify.

Live Presence

See who else is editing in real time — no more silent overwrites or "who changed this?" moments.

Annotation & Notes

Attach intent notes to nodes — document "why this prompt" for future maintainers.

One-Click Deploy

Publish approved version directly to API, Webapp, or embedded integration.

This matters most for teams larger than 2 people. Solo builders can ignore most of this. Ask: "Who here works on AI features with a team of 3+?" — then focus on the role & version points.
