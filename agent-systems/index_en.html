Gen 1 → Gen 2 transition happened fast (2022-2023). Gen 3 is where production teams are hitting friction today — compliance, repeatability, team scale. That's what Dify's latest releases directly address.

Three Architectural Advances

The features that define production-grade agent systems in Dify's latest releases.

Human-in-the-Loop Nodes
Dedicated execution nodes that pause a workflow and wait for human approval, override, or rejection before continuing. Enables auditable, compliance-friendly AI.
Agent × Skills

Sandboxed Runtime & Collaboration
Isolated code execution environment for Python / JS nodes, plus team-level workflow co-authoring with role-based editing and version history.
Each feature maps to a real production pain-point: HITL → compliance & audit trails Skills → code duplication & maintenance burden Sandbox + Collab → team scale & security

Human-in-the-Loop (HITL)

Most systems treat AI and humans as relay runners — one waits while the other works. Dify embeds human touchpoints as first-class citizens in the workflow graph itself.

Available in Workflow and Chatflow modes
Key message: "This isn't an external review process bolted on the side. It's a native node in Dify's execution graph — the architecture itself guarantees the pause and the resume."

Why Partial Automation Fails

Pure automation sacrifices quality control; pure human operation sacrifices efficiency. Three challenges sit at the center.

Shifting Objectives

Complex projects rarely follow a straight line. Objectives evolve, contexts change — AI workflows excel at predefined steps but can't adapt to mid-course pivots without structured human touchpoints.

Trust Gap

Organizations remain hesitant to fully automate high-stakes processes. Without visible checkpoints, AI adoption stalls — not because the technology fails, but because the system feels uncontrollable.

Integration Complexity

Existing workarounds require building separate approval queues, writing webhook callbacks, and manually threading state back — engineering overhead that makes human oversight feel like a punishment, not a feature.

Human Input Node

A native workflow node that pauses execution, surfaces context to a human, and routes onward based on their decision — no external system required.

Frame it as a product positioning story: we've seen what doesn't work (clunky external systems) and built something that feels native. Ask: "Which of these three resonates most with your current situation?"

Agent × Skills

You wouldn't inline a sort algorithm every time you need one. The same principle applies to LLM reasoning — Skills are the functions of agent design.

Works across Workflow and Agent modes
Transition: "Great — humans are in the loop. Now, how do we make the AI side more maintainable and scalable?"

The Monolithic Agent Problem

As agents grow, every new capability gets stuffed into the same prompt — until nothing is testable, versioned, or reusable.

Before — Monolithic Agent
Agent Prompt (500+ tokens)
search logic format rules API calls error handling output schema retry logic
Problems
duplicated across agents hard to test in isolation version drift
After — Skill-Based Agent
Agent (thin orchestrator)
goal context skill calls
Skill Library (shared)
search_web format_report call_api retry_safe
Analogy: Skills are like functions in software — you don't inline the sort algorithm every time. Same principle applies to LLM reasoning building blocks.

What Is a Skill?

A versioned, testable, composable execution unit with defined inputs and outputs.

Encapsulated

Self-contained logic with explicit input schema, output schema, and internal prompt.

Reusable

Published once, invoked from any Agent or Workflow across your workspace.

Testable

Run skills in isolation with fixture inputs — no need to trigger the full agent flow.

Versioned

Publish major/minor versions; agents pin to a version to avoid breakage on update.

Dynamic Variables

Skill inputs are assembled at runtime from conversation context, prior node outputs, or static config.

Composable

Chain multiple skills in sequence or fan-out in parallel inside a single agent step.

Demo cue: Open Skill Editor — show schema definition → publish → add to an agent → run. Highlight the variable assembly panel where you map context → skill inputs.

Skills live in the Execution Layer — agents in the Reasoning Layer. Separating these concerns is what makes large agent systems maintainable.
Scenario: Research & Report Agent

A single agent call composes web_search v2 → summarize_doc v1 → format_report v3 in sequence. Each skill was tested in isolation, version-pinned by the agent, and shared across three other agents in the workspace.

This layered view maps directly to how software architects think about service layers. Reasoning: "what to do?" — Execution: "how to do it?" The Skills model makes both layers independently testable and scalable. The scenario card makes it concrete — skills aren't theoretical, they're what you build once and reuse everywhere.

Sandboxed Runtime & Collaboration

AI workflows that run arbitrary code need hard isolation boundaries. And as teams grow, "one person edits the workflow" stops working.

Available in self-hosted and Dify Cloud
Transition: "The last piece: running code safely and letting teams work together."

The Risk of Unconstrained Code in AI Workflows

Code nodes unlock powerful transformations — but without isolation, every node is a potential attack surface.

Host System Access

Without isolation, a code node can read credentials, environment variables, or any file on the host — leaking sensitive data silently without any indication.

No Resource Limits

Runaway loops or memory-heavy operations block the entire worker process — one bad prompt can take down production for all users.

Supply Chain Risk

Code nodes can import third-party packages. A compromised package can silently send your workflow data to an external server. Without network restrictions, you'd never know.

Sandboxed Runtime solves this

Dify runs every code node in an isolated container — no host access, enforced CPU/memory limits, and a network allowlist. Enabled by default on Cloud; one flag in self-hosted.

This slide is most relevant to DevOps and platform engineers in the audience. The supply chain risk point often surprises people — they think about code they write, not the packages it imports.

Sandboxed Code Execution

Python and JavaScript nodes run in an isolated environment — no shared state, no host access.

Isolation Guarantees
  • No access to host filesystem
  • Network calls restricted by allowlist
  • Memory & CPU limits enforced per run
  • Execution timeout configurable per node
Supported Runtimes
Python 3.11+ JavaScript (Node 20) Custom packages via pip/npm

Collaborative Workflow Development

Multiple team members author, review, and ship workflows together — with roles and version history.

Role-Based Access

Viewer, Editor, and Owner roles per workspace. Non-owners can draft but not publish.

Version History

Every publish creates a named snapshot. Roll back in one click if a new version degrades performance.

Draft → Review → Publish

Git-like lifecycle: draft changes, request peer review, merge to live — all inside Dify.

Live Presence

See who else is editing in real time — no more silent overwrites or "who changed this?" moments.

Annotation & Notes

Attach intent notes to nodes — document "why this prompt" for future maintainers.

One-Click Deploy

Publish approved version directly to API, Webapp, or embedded integration.

Scenario: Platform Team at Scale

A data engineer drafts a new RAG node. A prompt engineer reviews and annotates the reasoning. The platform lead publishes v3 — with v2 still running in staging. If v3 degrades recall quality, one click rolls back — full history preserved.

This matters most for teams larger than 2 people. Solo builders can ignore most of this. Ask: "Who here works on AI features with a team of 3+?" — then focus on the role & version points. The scenario card bridges theory to practice for engineering leads.
