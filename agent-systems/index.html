第一部分

AI 系统的演进

从单轮 Prompt 迈向完全编排、可控的 Agent 架构。

过渡语："我们先聊聊为什么我们在这里 —— 以及什么变了。"

LLM 应用的三代演进

每一代都释放了新的价值 —— 也带来了新的复杂性。

第一代

Prompt → 响应

单轮补全。无记忆、无工具、无状态。

ChatGPT 封装 单次摘要
第二代

管道编排

节点链式调用，支持数据转换、RAG 与条件分支。

LangChain Dify Workflow
第三代 · 当下

Agent 系统

在人类输入处暂停。复用 Skills 与 SOP。显式传递交付物。在沙箱里安全执行。

HITL Skills 沙箱
第一代 → 第二代的过渡非常快（2022–2023）。 第三代是生产团队今天遇到摩擦的地方 —— 合规、可重复性、团队规模。 这正是 Dify 最新版本直接解决的问题。

三大架构演进

定义 Dify 生产级 Agent 方向的三项关键能力。

Human-in-the-Loop 节点
专用执行节点，暂停工作流并等待人工审批、修改或拒绝后再继续。支持可审计、合规友好的 AI 部署。
Agent × Skills
把 Agent 收敛为更薄的编排层，围绕可复用 Skills、SOP 化上下文和显式交付物来组织推理与执行。
沙箱运行时 & 协作开发
用 POSIX 风格运行时承载隔离执行与 Command 工作流，再把共享知识和团队协作带回工作流本身。
每个特性都对应一个真实的生产痛点： HITL → 合规 & 审计追踪 Skills → 代码重复 & 维护负担 沙箱 + 协作 → 团队规模 & 安全
02
第二部分

Human-in-the-Loop（HITL）

把人的判断放进工作流图里，而不是放在工作流外面。

可在 Workflow 和 Chatflow 模式中使用
HITL 在 Dify 里是原生节点，不是外挂审批系统。

为什么团队需要 HITL

最好的自动化，不是完全无人参与，而是只在人真正该介入的时刻介入。

目标会变化

工作进行到一半时，优先级和上下文可能会变，没有暂停点，流程就会变得僵硬。

信任仍然缺失

高风险流程里，团队需要看得见的检查点，才敢让 AI 代表业务继续执行。

集成过于复杂

外部审批队列和 Webhook 会把人工审核变成额外工程，而不是原生能力。

Human Input 节点

在图里暂停、拿到人的判断，然后沿正确分支原生恢复执行。

核心对比是原生暂停恢复和自建审批链路。

Human Input 节点如何工作

执行暂停，人拿到正确上下文，工作流按三种结果之一恢复。

编辑并批准

修正字段后，把新值送回执行流。

拒绝

进入回退分支、升级路径或终止处理。

这一页只讲暂停、审核、恢复三件事。

三个组成部分

每个 Human Input 节点都可以拆成三个简单部分：送达、绑定、路由。

送达

生成审核页，并把它送到正确的人手里。

变量

插入工作流字段，让人可编辑，再安全地把新值传回去。

动作

定义按钮、分支和超时规则，让流程始终知道如何恢复。

常见用法
审批动作 人工升级 补填表单信息
配置心智模型要短而稳定。

HITL 最适合放在哪里

好的 HITL 节点应该放在决策边界上，而不是铺满整条流程。

外部动作之前

在发送邮件、发布内容、提交工单或触达客户之前暂停一次。

置信度下降时

在异常、边界情况或低置信结果出现时再人工介入，而不是每次都审。

上下文缺失时

只补一个关键字段，然后让流程带着新值继续自动执行。

策略要求签核时

财务、合规和对客流程通常都需要一个看得见的审批点。

节点不一定要多，位置放对通常更重要。
重点讲放置策略，而不是数量。

如何设计审核体验

好的审核页应该做到：一个界面、正确上下文、明确动作。

一个界面

在 Webapp、邮件或聊天工具里直接完成，不要让审核者再回去翻工作流。

可编辑的上下文

只暴露人真正能修正的字段，不要把整份状态都扔给审核者。

明确动作

按钮要和真实分支一一对应，比如批准、请求修改、升级处理。

如果审核者还需要打开另一个系统才能完成任务，说明这个节点设计还没完成。
审核页应该是工作界面，而不只是通知入口。

梁 · 投资服务团队

当自动化结果要直接面向客户时，HITL 可以把专家判断插在最关键的位置。

扩张难题
40 分钟
每位客户的人工处理时间
100+
需要服务的客户数

报告生成已经自动化，但财务更新真正发给客户前，仍然需要合规团队做最后确认。

HITL 放置位置
汇总之后 异常出现时 发送之前

审核者看到的就是客户最终会看到的内容，必要时编辑后再一键批准。到六月，100 位客户都收到了一致的高质量报告。

“系统负责计算，人类在最关键的地方施加判断。”
这是审批和合规场景。

HITL 不只是审批，它也可以成为补齐缺失上下文的最干净方式。

支持难题

员工要在 HR、财务和 IT 的不同门户间来回切换。很多请求一开始就缺少路由所需的关键信息。

“结果像是一种协作智能。”
这个案例说明 HITL 也是补信息节点。

第三部分

Agent × Skills

Agent 会变得更薄：选择正确的 SOP，调用正确的 Skill，并留下可复用的交付物。

适用于 Workflow 和 Agent 模式
这一段讲的是怎样让 AI 侧更容易维护，也更容易交接。

为什么今天的 Agent 仍然不够好

今天的高级团队，依然常常在绕着运行时做事，而不是直接依赖它。

Agent 流程仍偏单次执行

多轮行为和多 Agent 交接，在 Workflow 里仍然不够自然。

工具菜单制造噪音

工具一多，Prompt 空间就被挤占，工具路由也会越来越脆弱。

文件交接很脆弱

文件经常依赖隐藏标识符传递，而不是依赖直观的运行时工件。

调试链路太长

策略、工具路由、文件流和记忆分散在多层，出错时很难定位。

核心问题不只是模型不够强，而是运行方式不够自然。

从单体 Prompt 到薄编排层

Agent 应该负责决策，而不是把所有执行细节都塞进 Prompt。

之前 —— Prompt 什么都管
Prompt 里塞着
工具路由 文件处理 重试逻辑 输出格式
容易出的问题
逻辑重复 难以测试 工具膨胀 状态隐藏
之后 —— Agent 只做编排
Agent 负责
目标 选择 SOP 调用 Skills 挑选交付物
Workflow 拿到
文本 文件 结构化字段 记忆快照
Skill 对 Agent 的意义，类似函数对软件的意义。

交付物缺失会让 Workflow 失灵

如果真正有价值的产物还停留在 Agent 内部记忆里，下游节点就只能从自然语言里猜。

示例：IF/ELSE 节点只能从文本里推断状态。

文本不是状态

靠判断 Agent 有没有说出 success，既脆弱，也难维护。

文件会消失

原始表格、报告或中间文件可能埋在记忆里，下游只看得到总结。

Agent 无法接力

后一个 Agent 不能稳定知道前一个 Agent 究竟交付了什么。

下游用不了的东西，就不算真正进入了 Workflow。

一个节点应该交付什么

生产级工作流需要的，不只是一个润色后的答案。

文本回答

给人看的最终解释或回复。

文件

报告、表格、图片等，可继续被下游节点使用。

结构化字段

状态、决策、ID、参数等，供分支或工具直接读取。

记忆快照

让后续节点可以从中提取事实、参数甚至文件。

如果下游不能消费它，那它就不算真正的交付物。
这是目标中的交付面。

什么是 Skill？

Skill 是一个可复用的执行单元，把 SOP 上下文、运行行为和稳定的交接契约打包在一起。

由 SOP 驱动

“这类事情怎么做”不再分散在各个节点，而是跟 Skill 一起沉淀。

可复用

发布一次，就可以被不同 Agent 和 Workflow 反复调用。

可测试

用固定输入单独运行 Skill，而不必触发整条流程。

可锁版本

Agent 可以锁定稳定版本，而不是每次共享 Skill 更新都被动受影响。

典型输入来源
对话上下文 前置节点输出 文件 记忆提取
Skill 是可复用的上下文加可复用的执行。

一套 SOP，多种入口

Context Engineering 需要一个共享归宿，而不是重复粘贴 Prompt。

今天 —— SOP 埋在节点里
同一 SOP 重复写 难以评审 最佳实践漂移
更好的方式 —— 共享 /sops 工作区
写一次 按入口文件区分 随工作流版本化
很多 Agent 共享的是同一套 SOP，只是从不同文件进入、关注范围不同。
这是把 Context Engineering 变成团队基础设施。

Skill + SOP Agent 架构

推理层保持轻量，执行层围绕文件、命令和可复用交付物展开。

输入
用户请求 前置节点输出 上传文件
Agent 层
选择 SOP 组装上下文 调用 Skills 决定下一步
工作区
/sops 命令 文件 版本化 Skills
交接物
文本 文件 字段 记忆 HITL
更好的心智模型是 SOP、文件和工件，而不是一个越来越大的工具下拉框。
关键是工作区心智模型，而不是工具菜单心智模型。

记忆提取让上下文可复用

Memory 不再只是实现细节，而是可以继续流转的工作流工件。

每个节点保留自己的记忆

文本不再是唯一输出，节点的工作上下文也会被保留下来。

后续节点按需提取

事实、参数，甚至文件，都可以在后续步骤里从记忆中重新提取出来。

上下文变成可复用状态

关键产物不再消失在对话历史里，而是能够继续向下游流转。

这一页解释的是怎样把隐藏上下文变成可复用状态。

第四部分

沙箱运行时 & 协作开发

当 Agent 围绕 SOP、文件和显式交付物工作时，运行时必须同时做到好用和安全。

适用于自托管和 Dify Cloud
这一段讲的是可用的运行时和围绕它的协作产品面。

Command 节点：一个很小但很强的原语

输入一行命令，输出 stdout，把文件留在运行时里交给下一步。

示例
report --input ./turnsheet.csv --format json
命令行输入 stdout 输出 文件留在运行时
对 LLM 更自然

模型在预训练里已经见过大量命令、管道和文件路径。

产品面更小

不需要为每个小转换动作都单独设计一套工具 UI。

交接更清晰

大的工件保留成文件，显式地往下传，而不是硬塞回 Prompt。

Command 节点很小，但会打开一种更自然的运行时模型。

从工具列表到 POSIX 工作区

不要把每种能力都做成一张 Tool 卡，而是让运行时直接暴露命令、文件和 stdout。

之前 —— Tool-first 编排
step1: A = google_search(query="Dify", max_size=30)
step2: B = summary(query=A)
类型转换隐藏 输出停在内存里 每个工具都要做 UI
之后 —— POSIX 风格执行
summary --query "$(google_search --query dify --max_size 30)"
统一字符串接口 shell 原生组合 先 ls /bin 认识环境
运行时对人更简单，对 LLM 也更可读。

无约束代码的风险

Agent 需要真实执行面，但不应该直接拿到宿主机。

宿主系统访问

没有隔离时，代码可以读取本地凭证、环境变量和文件。

没有资源边界

失控循环或内存暴涨会拖垮共享运行时上的所有任务。

供应链风险

导入的第三方包可能静默外传工作流数据，除非环境被严格控制。

沙箱的目标不是削弱能力，而是把能力安全地关进边界里。
关键不是减少能力，而是控制能力边界。

沙箱代码执行

给 Agent 一个像小型 Unix 工作站一样的环境，同时包在严格的安全边界内。

安全边界
  • 无法访问宿主文件系统
  • 网络受白名单限制
  • 每次运行都有 CPU / 内存上限
  • 每个节点可配置超时
可用运行时表面
ls /bin stdin/stdout I/O 文件作为交接 Python 3.11+ JavaScript (Node 20) 外置文件存储
如何启用
Cloud 版默认开启 自托管设置 SANDBOX=true
这就是让命令执行安全进入生产环境的前提。

协作工作流开发

工作流本身会变成团队共同维护的产品表面。

角色访问控制

不同成员可以分别起草、审查、发布，而不会互相覆盖。

版本历史

每次发布都会形成快照，便于对比和快速回滚。

草稿 → 审查 → 发布

流程生命周期变得清晰可重复，而不是散落在截图和聊天记录里。

共享 SOP 资产

最佳实践不再是个人 Prompt 片段，而会沉淀成团队资产。

一个简单团队流

一个人起草工作流，另一个人审查 SOP，负责人发布通过版本，同时完整保留历史。

当工作流变成团队资产后，协作就不再是附加功能。

融合全景

一个生产级 Agent 系统：推理、运行时执行和人工审核都围绕显式交付物协同工作。

输入

交付物 + 可观测性
文本响应 生成文件 结构化字段 记忆快照 追踪日志 成本追踪
生产级的关键，是每一步都要留下下一步真正能用的产物。
这张架构图可以在向工程负责人提议引入时分享。 强调："不需要从第一天就用齐三个 —— 单独的 HITL 就能解锁受监管部署；单独的 Skills 就能减少维护负担。"
