第一部分

AI 系统的演进

从单轮 Prompt 迈向完全编排、可控的 Agent 架构。

过渡语："我们先聊聊为什么我们在这里 —— 以及什么变了。"

LLM 应用的三代演进

每一代都释放了新的价值 —— 也带来了新的复杂性。

第一代

Prompt → 响应

单轮补全。无记忆、无工具、无状态。

ChatGPT 封装 单次摘要
第二代

管道编排

节点链式调用，支持数据转换、RAG 与条件分支。

LangChain Dify Workflow
第三代 · 当下

Agent 系统

基于人工输入暂停、恢复、分支。可复用技能。沙箱执行。团队协作。

HITL Skills 沙箱
第一代 → 第二代的过渡非常快（2022–2023）。 第三代是生产团队今天遇到摩擦的地方 —— 合规、可重复性、团队规模。 这正是 Dify 最新版本直接解决的问题。

三大架构演进

定义 Dify 最新版本中生产级 Agent 系统的核心特性。

Human-in-the-Loop 节点
专用执行节点，暂停工作流并等待人工审批、修改或拒绝后再继续。支持可审计、合规友好的 AI 部署。
Agent × Skills
封装化、可复用的技能模块，将推理与执行解耦。Agent 动态组合技能，减少重复代码，实现可扩展的多步骤 Agent 设计。
沙箱运行时 & 协作开发
隔离的 Python/JS 代码执行环境，以及支持角色权限与版本历史的团队级工作流协同创作。
每个特性都对应一个真实的生产痛点： HITL → 合规 & 审计追踪 Skills → 代码重复 & 维护负担 沙箱 + 协作 → 团队规模 & 安全
第二部分

Human-in-the-Loop（HITL）

可控、可审计的 AI —— 工作流在人工决策时暂停，决策后恢复。

可在 Workflow 和 Chatflow 模式中使用
核心信息："AI 会出错。HITL 是让它能在受监管环境中部署的安全网。"

完全自主 AI 的问题

真实合规场景需要人工检查点 —— 而不是"相信 AI"。

⚠️

无审计追踪

全自主管道产生的输出没有记录谁做了什么决定，使合规报告无从完成。

🔁

错误无声级联

中间步骤的错误输出在任何人注意到之前已向下游传播 —— 有时触发不可逆的副作用。

🔒

监管障碍

金融、医疗、法律类 AI 部署要求在关键决策节点留有人工确认的书面记录。

✅

HITL 节点解决这一问题

在工作流任意位置插入暂停关卡。人工审查上下文后批准或修改，执行恢复并留有完整追踪。

具体行业：医疗报告生成、合同起草、金融异常审核、内容审核队列。

HITL 节点如何工作

1
触发
工作流运行到 HITL 节点边界
2
暂停
执行挂起；上下文 + LLM 输出呈现给审核人
3
决策
人工批准、编辑或拒绝，可附加备注
4
恢复
工作流继续，人工决策记录入追踪日志
决策结果
  • 批准 —— 沿用 AI 输出继续执行
  • 编辑并批准 —— 修改内容后继续
  • 拒绝 —— 跳转备用分支或终止
审计记录
  • 审核人身份 & 时间戳
  • AI 原始输出 vs. 最终值
  • 决策备注 / 理由
演示提示：在 Dify 画布中展示 HITL 节点 → 触发运行 → 打开审批收件箱 → 批准 → 展示追踪。 强调：工作流是真正暂停的 —— 默认不会超时（可配置）。

HITL 落地实践

跨合规敏感行业的真实部署模式。

合同审核

LLM 起草条款摘要；法务团队在交付客户前批准。

医疗分诊

AI 建议紧急程度评分；医生在转诊前确认。

金融风控

异常检测标记交易；分析师批准升级或关闭告警。

内容审核

低置信度分类输出在执行前等待人工判断。

外发沟通

AI 起草客户邮件；管理者批准后批量发送。

第三部分

Agent × Skills

模块化、可复用的技能式 Agent 设计 —— 大规模解耦推理与执行。

适用于 Workflow 和 Agent 模式
过渡语："好的 —— 人已经在环路中了。现在，怎样让 AI 侧更易维护、更可扩展？"

单体 Agent 问题

随着 Agent 增长，它们变得错综复杂、难以测试、无法复用。

之前 —— 单体 Agent
Agent Prompt（500+ tokens）
搜索逻辑 格式规则 API 调用 错误处理 输出 Schema 重试逻辑
问题
跨 Agent 重复 难以隔离测试 版本漂移
之后 —— 技能式 Agent
Agent（薄编排层）
目标 上下文 技能调用
技能库（共享）
search_web format_report call_api retry_safe
类比：技能就像软件中的函数 —— 你不会每次都内联排序算法。 同样的原则适用于 LLM 推理构建块。

什么是 Skill？

带有明确输入输出定义的、可版本化、可测试、可组合的执行单元。

封装

具有明确输入 Schema、输出 Schema 和内部 Prompt 的自包含逻辑。

可复用

发布一次，可从工作区内任何 Agent 或 Workflow 调用。

可测试

使用固定输入隔离运行技能 —— 无需触发完整的 Agent 流程。

可版本化

发布主/次版本；Agent 锁定特定版本以避免更新后的破坏性变更。

动态变量

技能输入在运行时从对话上下文、前置节点输出或静态配置中组装。

可组合

在单个 Agent 步骤内顺序链接或并行扇出多个技能。

演示提示：打开技能编辑器 —— 展示 Schema 定义 → 发布 → 添加到 Agent → 运行。 重点演示将上下文映射到技能输入的变量组装面板。

技能位于执行层，Agent 位于推理层。分离这两个关注点，是让大型 Agent 系统可维护的关键。
这个分层视图与软件架构师思考服务层的方式直接对应。 推理："做什么？" —— 执行："怎么做？" Skills 模型使两层都可独立测试和扩展。
第四部分

沙箱运行时 & 协作开发

安全代码执行 + 团队级工作流协同创作 —— 生产基础设施层。

适用于自托管和 Dify Cloud
过渡语："最后一块拼图：安全运行代码并让团队协同工作。"

沙箱代码执行

Python 和 JavaScript 节点在隔离环境中运行 —— 无共享状态，无宿主访问。

隔离保证
  • 无法访问宿主文件系统
  • 网络调用受白名单限制
  • 每次运行强制内存 & CPU 限制
  • 每个节点可配置执行超时
支持的运行时
Python 3.11+ JavaScript (Node 20) 通过 pip/npm 自定义包
code_node.py 
# 在隔离沙箱中运行 — 无宿主访问
def main(inputs: dict) -> dict:
    text = inputs["content"]
    words = len(text.split())
    # 自定义转换逻辑
    summary = text[:200] + "..."
    return {
        "summary": summary,
        "word_count": words
    }
自托管部署：在 docker-compose 中启用 SANDBOX 服务以激活隔离。云端版本默认启用沙箱。
核心信息："你现在可以在工作流中编写任意转换代码，而不会打开安全漏洞。" 演示提示：添加 Code 节点，粘贴 Python 函数，运行 —— 在追踪中展示输出。

协作工作流开发

多名团队成员共同编写、审查和发布工作流 —— 有角色权限和版本历史。

角色访问控制

工作区内的查看者、编辑者和所有者角色。非所有者可起草但无法发布。

版本历史

每次发布创建命名快照。新版本性能下降时一键回滚。

草稿 → 审查 → 发布

类 Git 生命周期：起草变更、发起同行审查、合并到线上 —— 全在 Dify 内完成。

实时在线状态

实时看到谁在编辑 —— 不再有无声覆盖或"谁改了这里？"的困扰。

注释 & 备注

在节点上附加意图备注 —— 为未来维护者记录"为什么这样写 Prompt"。

一键部署

将审批通过的版本直接发布到 API、Web 应用或嵌入式集成。

这对超过 2 人的团队最有价值。独立开发者可以忽略大部分内容。 提问："这里有谁和 3+ 人团队一起做 AI 功能？" —— 然后重点讲角色和版本部分。
