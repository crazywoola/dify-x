每一代都释放了新的价值 —— 也带来了新的复杂性。

第一代

ChatGPT 封装 单次摘要
第二代

管道编排

节点链式调用，支持数据转换、RAG 与条件分支。

LangChain Dify Workflow
第三代 · 当下

Agent 系统

基于人工输入暂停、恢复、分支。可复用技能。沙箱执行。团队协作。

HITL Skills 沙箱
第一代 → 第二代的过渡非常快（2022–2023）。 第三代是生产团队今天遇到摩擦的地方 —— 合规、可重复性、团队规模。 这正是 Dify 最新版本直接解决的问题。

三大架构演进

定义 Dify 最新版本中生产级 Agent 系统的核心特性。

Human-in-the-Loop 节点
专用执行节点，暂停工作流并等待人工审批、修改或拒绝后再继续。支持可审计、合规友好的 AI 部署。
Agent × Skills
封装化、可复用的技能模块，将推理与执行解耦。Agent 动态组合技能，减少重复代码，实现可扩展的多步骤 Agent 设计。
沙箱运行时 & 协作开发
隔离的 Python/JS 代码执行环境，以及支持角色权限与版本历史的团队级工作流协同创作。
每个特性都对应一个真实的生产痛点： HITL → 合规 & 审计追踪 Skills → 代码重复 & 维护负担 沙箱 + 协作 → 团队规模 & 安全
02
第二部分

Human-in-the-Loop（HITL）

大多数系统把 AI 和人类当作接力跑手 —— 一方等待另一方工作。Dify 将人工介入点作为工作流图的一等公民，原生嵌入执行架构本身。

可在 Workflow 和 Chatflow 模式中使用
核心信息："这不是外挂的审核流程，而是 Dify 执行图中的原生节点 —— 架构本身保证了暂停与恢复。"

为什么半自动化会失败

纯自动化牺牲质量管控；纯人工操作牺牲效率。三大挑战横亘其中。

目标漂移

复杂项目鲜少一路向前。目标演变、上下文变化 —— AI 工作流擅长预定步骤，却无法在没有人工介入节点的情况下适应中途转向。

信任缺口

组织对全面自动化高风险流程依然心存顾虑。没有可见的检查点，AI 落地就会停滞 —— 不是因为技术失败，而是因为系统感觉失控。

集成复杂度

现有方案（LangGraph interrupt、go-to-human）使用繁琐 —— 过度技术化、难以可视化，非技术团队成员维护成本高。

Human Input 节点

原生工作流节点，暂停执行、向人类呈现上下文，并根据决策路由后续流程 —— 无需外部系统。

以产品定位故事为框架：我们见过什么不奏效（笨重的外部系统），并构建了感觉原生的解决方案。 提问："这三个挑战中，哪个与你当前情况最相关？"

Human Input 节点

一个带有三个可配置组件的暂停关卡 —— 内容如何送达、人类看到什么、他们可以做什么。

1
送达配置
自动生成审核页面，通过邮件、Slack 或 Dify Webapp 送达 —— 每个节点独立选择渠道。
2
变量引用
使用斜杠快捷键在页面中嵌入任意工作流变量。人类可读取、编辑，修改后的值回传执行流。
3
动作定义
配置带标签的按钮（如：批准、请求修改、升级）。每个按钮将执行路由到不同下游路径，支持设置超时规则。
送达渠道
Dify Webapp 邮件 Slack（通过插件） 自定义 Webhook
路由结果
编辑后继续 按决策分支 超时回退 完整审计追踪
演示提示：在 Dify 画布中展示 Human Input 节点 —— 配置送达渠道（邮件），用斜杠命令添加变量引用，设置两个动作按钮（批准 / 请求修改）并各自连接不同下游路径。 强调：超时规则意味着无人值守的工作流不会永远阻塞。

当研发部 Jason 询问报销问题时，系统检测到缺少地点信息，通过 Human Input 节点向他补充收集，随后精准送达上海办公室的相关政策。

"我们在构建将 AI 的一致性与人类输入的适应性相结合的工作流。结果感觉像是协作智能。"
这些是真实的客户场景，并非假设。 梁的故事展示了战略性节点放置；闵的故事展示了上下文 Human Input（不仅仅是审批，而是流程中途收集数据）。 提问："哪种模式更接近你要构建的场景？"

第三部分

Agent × Skills

你不会每次需要排序时都把算法内联进去。同样的原则适用于 LLM 推理 —— Skill 就是 Agent 设计中的函数。

适用于 Workflow 和 Agent 模式
过渡语："好的 —— 人已经在环路中了。现在，怎样让 AI 侧更易维护、更可扩展？"

单体 Agent 问题

随着 Agent 增长，每个新能力都被塞进同一个 Prompt —— 直到没有任何东西可测试、可版本化、可复用。

之前 —— 单体 Agent
Agent Prompt（500+ tokens）
搜索逻辑 格式规则 API 调用 错误处理 输出 Schema 重试逻辑
问题
跨 Agent 重复 难以隔离测试 版本漂移
之后 —— 技能式 Agent
Agent（薄编排层）
目标 上下文 技能调用
技能库（共享）
search_web format_report call_api retry_safe
类比：技能就像软件中的函数 —— 你不会每次都内联排序算法。 同样的原则适用于 LLM 推理构建块。

什么是 Skill？

带有明确输入输出定义的、可版本化、可测试、可组合的执行单元。

封装

具有明确输入 Schema、输出 Schema 和内部 Prompt 的自包含逻辑。

可复用

发布一次，可从工作区内任何 Agent 或 Workflow 调用。

可测试

使用固定输入隔离运行技能 —— 无需触发完整的 Agent 流程。

可版本化

发布主/次版本；Agent 锁定特定版本以避免更新后的破坏性变更。

动态变量

技能输入在运行时从对话上下文、前置节点输出或静态配置中组装。

可组合

在单个 Agent 步骤内顺序链接或并行扇出多个技能。

演示提示：打开技能编辑器 —— 展示 Schema 定义 → 发布 → 添加到 Agent → 运行。 重点演示将上下文映射到技能输入的变量组装面板。

技能位于执行层，Agent 位于推理层。分离这两个关注点，是让大型 Agent 系统可维护的关键。
场景：研究与报告 Agent

单次 Agent 调用依次组合 web_search v2 → summarize_doc v1 → format_report v3。每个技能经过隔离测试、由 Agent 锁定版本，并在工作区内的另外三个 Agent 中共享复用。

这个分层视图与软件架构师思考服务层的方式直接对应。 推理："做什么？" —— 执行："怎么做？" Skills 模型使两层都可独立测试和扩展。 场景卡让抽象变得具体 —— 技能不是理论，而是构建一次、处处复用的东西。
04
第四部分

沙箱运行时 & 协作开发

运行任意代码的 AI 工作流需要硬性隔离边界。而随着团队扩大，"一个人编辑工作流"的模式也不再可行。

适用于自托管和 Dify Cloud
过渡语："最后一块拼图：安全运行代码并让团队协同工作。"

AI 工作流中无约束代码的风险

代码节点解锁了强大的转换能力 —— 但没有隔离，每个节点都是潜在的攻击面。

任意文件系统访问

没有隔离时，代码节点可以读取宿主机上的凭证、环境变量或任意文件 —— 成为静默的数据泄露通道。

无资源限制

失控的循环或大内存操作会阻塞整个 Worker 进程 —— 一个错误的 Prompt 可能让所有用户的生产环境宕机。

供应链风险

运行时拉取的 pip 和 npm 依赖可能包含恶意代码。没有网络限制，被入侵的包会悄无声息地回传数据。

沙箱运行时解决这一问题

Dify 在隔离容器中运行每个代码节点 —— 无宿主访问、强制 CPU/内存限制、网络白名单。Cloud 版本默认启用；自托管只需一个配置项。

这张幻灯片对观众中的 DevOps 和平台工程师最有价值。 供应链风险这点往往让人出乎意料 —— 他们考虑的是自己写的代码，而不是它导入的包。

沙箱代码执行

Python 和 JavaScript 节点在隔离环境中运行 —— 无共享状态，无宿主访问。

隔离保证
  • 无法访问宿主文件系统
  • 网络调用受白名单限制
  • 每次运行强制内存 & CPU 限制
  • 每个节点可配置执行超时
支持的运行时
Python 3.11+ JavaScript (Node 20) 通过 pip/npm 自定义包

协作工作流开发

多名团队成员共同编写、审查和发布工作流 —— 有角色权限和版本历史。

角色访问控制

工作区内的查看者、编辑者和所有者角色。非所有者可起草但无法发布。

版本历史

每次发布创建命名快照。新版本性能下降时一键回滚。

草稿 → 审查 → 发布

类 Git 生命周期：起草变更、发起同行审查、合并到线上 —— 全在 Dify 内完成。

实时在线状态

实时看到谁在编辑 —— 不再有无声覆盖或"谁改了这里？"的困扰。

注释 & 备注

在节点上附加意图备注 —— 为未来维护者记录"为什么这样写 Prompt"。

一键部署

将审批通过的版本直接发布到 API、Web 应用或嵌入式集成。

场景：规模化平台团队

数据工程师起草一个新的 RAG 节点。Prompt 工程师审查并添加推理注释。平台负责人发布 v3 —— v2 仍在预发布环境运行。如果 v3 导致召回质量下降，一键回滚 —— 完整历史永久保留。

这对超过 2 人的团队最有价值。独立开发者可以忽略大部分内容。 提问："这里有谁和 3+ 人团队一起做 AI 功能？" —— 然后重点讲角色和版本部分。 场景卡为工程负责人搭起了理论到实践的桥梁。
