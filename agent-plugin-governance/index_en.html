TRUST / COMPOSABILITY
This talk focuses on two practical questions: what governance is missing before open plugins enter enterprise environments, and how the new capabilities in 1.15 and 1.16 change the division of work between people and agents.
01 · Define the problem

Installing a plugin grants three kinds of access.

{ }

Code execution

Run dependencies, commands, binaries, and model-call logic at runtime.

↕

Data processing

Read files, conversations, knowledge, credentials, and workspace context.

◎

External connectivity

Reach third-party APIs, arbitrary URLs, remote services, and downstream processors.

Governance therefore has to cover rights, evidence, and accountability—not just whether a package installs.

From an enterprise perspective, a plugin is more than a feature. It receives execution, data-processing, and connectivity access. The rest of the governance design follows from those three rights.

Governance stack

Machines hold the floor, people judge context, and process handles disputes.

LAYER 01

Machine gates

Hard rejects, tier thresholds, signatures, SBOM, dependency and static scanning.

Goal: make the minimum quality bar deterministic.

LAYER 02

Human judgment

L3 deep review, dynamic testing, security exceptions, and dual sign-off.

Goal: handle context and intent machines cannot.

LAYER 03

Legal process

Author CoC, agreement alignment, notice, response, decision, appeal.

Goal: make enforcement explainable and reviewable.

Automation alone creates rigidity and false positives; manual review alone creates inconsistency and queues. Machines hold the floor, humans handle context, and procedure protects long-term legitimacy.

Trade-offs · govern governance

Stricter rules have a cost, so the buffers need to be designed with them.

FRICTION

Over-strict gates suppress contribution

Keep L1 frictionless; pilot two weeks in the official repo; use warning-only community rollout for 30 days.

FALSE POSITIVE

False positives and exceptions

Explain findings, link docs, support inline waivers, and require ≥2 maintainers for security exceptions.

MIGRATION

Existing author cost

Apply new rules first to new submissions; visualize legacy scan results and allow 90 days for Critical fixes.

DUE PROCESS

Takedown disputes

Notice → response → decision → appeal; security emergencies may remove first but must explain later.

CI may grow from roughly 3 minutes to 6–8; parallel stages and artifact reuse keep the cost bounded.

Source: proposal §5 embedded risk-and-mitigation table.
The research proposes a two-week official-repo pilot, thirty warning-only days for the community repo, documented waivers, two-maintainer security exceptions, and a ninety-day window for legacy Critical findings. Stricter rules need clearer paths to comply and appeal.

Target operating model

A production system has at least four distinct responsibilities.

01 / HUMAN

Intent & accountability

Set goals, approve high-risk actions, handle exceptions, own outcomes.

02 / AGENT

Exploration & execution

Decompose tasks, call Skills, and deliver structured artifacts in a controlled runtime.

03 / WORKFLOW

Orchestration & evidence

Manage state, sequence, HITL, retries, archives, cost, and observability.

04 / GOVERNANCE

Rights & boundaries

Plugin tiers, policy as code, identity, data flow, signing, monitoring, and due process.

Collaboration becomes operable when rights, state, and accountability each have an owner.

Humans own intent and accountability. Agents explore and execute. Workflows own state and evidence. Governance owns rights and boundaries. Every production system must answer all four.
