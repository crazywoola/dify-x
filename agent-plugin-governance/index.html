Governance before scale

TRUST / COMPOSABILITY
今天不按安全工具逐个介绍。我更想回答两个实际问题：开放插件进入企业环境前，需要补上哪些治理环节；1.15 和 1.16 出现的新能力，会怎样改变人与 Agent 的分工。
01 · 先定义问题

企业安装一个插件，实际上开放了三类权限。

{ }

代码运行权

在运行时执行依赖、命令、二进制与模型调用逻辑。

↕

数据处理权

读取文件、对话、知识库、凭证与工作区上下文。

◎

外部连接权

访问第三方 API、任意 URL、远端服务与下游处理者。

治理要围绕权限、证据和责任链展开，不能只看上架时有没有报错。

从企业视角看，插件不只是一个功能入口。它拿到的是代码执行、数据处理和外部连接权限。后面的治理设计都从这三类权限出发。

Six lenses · 六种视角

同一个插件，六个团队会问六类不同的问题。

01 / SECURITY

供应链安全

密钥、二进制、依赖漏洞、运行时行为与上架后变质。

02 / LEGAL

法务与合规

平台责任、子处理者、知识产权、下架与申诉程序。

03 / PRODUCT

产品信任

徽章、能力清单、数据流、签名、SBOM 与风险提示。

04 / ENGINEERING

工程效率

本地预检、分级 CI、可定位报告、artifact 复用与自动标签。

05 / COMMUNITY

社区可持续

低风险零摩擦、规则可预演、例外透明、处置有程序。

06 / BUSINESS

商业准入

让 Enterprise 安全团队能够评估、签字、持续复核。

安全团队看供应链，法务看责任链，产品要把信任信息做给用户看，工程和社区关心规则能否长期执行，商业团队最终要回答企业客户能不能通过安全评估。
Trust gap · 信任落差

如果对外说“更安全、更可靠”，运营证据必须跟得上。

Marketing language
“Safer, more reliable”

用户听到的是 publisher 风格的信赖承诺。

Operational substance
解包 + manifest 校验

当前运营更像 hosting service，仍缺安全扫描与持续监测。

主站已有 SOC 2、ISO 27001 与 GDPR 体系，插件市场不能成为尽调空白区。

来源：飞书提案 §2.4 与 §3.1。法律定位需由适用法域下的专业法务最终判断。
主站的合规体系会提高客户对插件市场的预期。平台到底更像 hosting 还是 publisher，也不只由条款决定；审核深度、推荐方式、徽章和营销措辞都会成为判断材料。过渡期文案是否调整，需要法务明确拍板。
Responsibility chain · 责任链

企业评估时会追问：数据去了哪里，出了问题谁负责。

01 · CUSTOMER

业务用户 / Controller

决定处理目的，承担内部授权与业务结果责任。

02 · PLATFORM

Dify / Processor?

承载执行、凭证、分发与审计；角色随部署形态变化。

03 · AUTHOR

插件作者 / Sub-processor?

决定实现与下游调用；协议与告知链仍需补强。

04 · DOWNSTREAM

第三方 API / 再处理者

数据流向、区域、保留策略与再委托需要透明。

Privacy

类型、目的、第三方。

IP & License

权属、许可、分发权。

Due process

通知、申辩、复议。

Evidence

签名、SBOM、时间戳。

来源：飞书提案 §3.2–§3.5。图中问号表示角色需按具体处理场景与合同关系认定。
不要把 GDPR 的角色关系说死，因此保留问号。治理要提供足够信息，让法务能判断：谁决定目的、谁实际处理、谁把数据交给谁，以及出了问题怎样通知、申诉和举证。
Risk proportionality · 风险分级

低风险保持顺滑，高风险再增加审核成本。

档位典型行为机器检查人工审核用户感知
L1固定 HTTPS API；无本地副作用硬拒绝 + Semgrep 警告抽样 20%Verified Basic · 零额外打断
L2用户文件 / URL；上传下载与富文本解析依赖 Medium+ 阻断；警告需说明全量轻量 reviewStandard · 数据流可见
L3命令、SQL、SSH、浏览器、本地文件、任意 HTTPwarning → error；Low+ 依赖阈值深度 review + 动态测试High-trust · 安装二次确认

原则：低风险不增摩擦，高风险不靠运气。

来源：飞书提案附录 C《风险分级标准》及其嵌入表格。
这页是整个治理体系的核心杠杆。只有分级，才能同时保护安全和社区活力。自评与机器结果取更高档，reviewer 可以调档；鼓励作者诚实披露，而不是让所有贡献者都承受 L3 成本。
Lifecycle governance · 全生命周期

一次 CI 通过只能说明当时；治理还要覆盖安装与后续更新。

01

作者

严格模板、.difyignore、风险自评、能力与隐私声明。

02

构建

源码 URL、commit SHA、锁依赖与可追溯包指纹。

03

入库 CI

密钥、二进制、依赖、SAST、manifest 与风险档位。

04

上架

官方 / Partner / 社区分层，展示徽章与可信信息。

05

安装运行

L3 二次确认；最小能力；核对沙箱与出站行为。

06

持续监测

新 CVE、异常调用、降级、SLA、通知与复议。

把治理画成生命周期，才能避免“CI 通过就永远可信”的错觉。版本可以变质、依赖会出现新漏洞、作者凭证会被盗，因此上架后监测和处置程序与入库扫描同等重要。

Governance stack · 三层治理

机器守底线，人判断语境，程序处理争议。

LAYER 01

机器门禁

硬拒绝、分级阈值、签名、SBOM、依赖与静态扫描。

目标：把最低质量线变成确定性。

LAYER 02

人工判断

L3 深度 review、动态测试、安全例外与双人 sign-off。

目标：处理机器看不懂的语境与意图。

LAYER 03

法律程序

Author CoC、协议衔接、通知、申辩、决定、复议与安全例外。

目标：让处置可解释、可执行、可复核。

纯自动化会制造误报与僵化，纯人工会制造不一致和积压。三层治理把不同类型的判断放到最合适的位置：机器守底线，人处理语境，程序保护长期关系。

Trade-offs · 治理也要被治理

更严格的规则会带来成本，缓冲机制要一起设计。

FRICTION

门禁过严抑制贡献

L1 零摩擦；官方仓试运行 2 周再下沉；社区仓前 30 天 warning 不阻断。

FALSE POSITIVE

误报与例外

命中说明 + docs；支持内联豁免；安全例外需 ≥2 maintainer 与业务理由。

MIGRATION

现有作者迁移成本

新规则优先作用于新提交；存量先做可视化扫描，Critical 项给 90 天修复窗口。

DUE PROCESS

封禁与下架争议

通知 → 申辩 → 决定 → 复议；安全例外可先下后告，但须事后说明。

CI 可能从约 3 分钟增至 6–8 分钟；用并行阶段和 artifact 复用控制成本。

来源：飞书提案 §5《实施风险与权衡》嵌入表格。
权衡表给出的做法很具体：官方仓先试两周，社区仓前三十天只告警；误报要有文档和豁免；安全例外至少两名 maintainer 批准；存量 Critical 给九十天窗口。规则越严格，越要让作者提前知道怎样通过和怎样申诉。

Feature spotlight · AI 自动生成应用

一句需求，先变成可检查的计划，再生成可编辑的 Workflow / Chatflow。

01 · UNDERSTAND

理解需求

结合工作区上下文给出可构建建议，并判断应用模式。

02 · PLAN FIRST

先展示节点计划

画布生成前先看真实节点大纲，范围不对可以及时停下。

03 · BUILD + CHECK

并行生成并验证

配置节点，验证工具和连接；失败时指出受影响节点。

04 · APPLY / REFINE

应用后继续修改

/refine 汇总新增、删除与变更，结果仍可编辑。

从“画每个节点”转向定义目标、审阅计划、处理例外生成物仍进入原有治理流程

这里的“自动生成应用”具体指 Workflow 和 Chatflow。用户在 Command K 中输入 /create，可以直接描述需求；系统先判断模式并给出真实节点计划，再生成和验证画布。/refine 会给出图差异摘要。它改变的不是谁拥有应用，而是谁负责第一稿：机器负责把自然语言编译成可编辑流程，人负责目标、计划审阅、工具授权和例外。当前功能仍属于 preview，Create-App 对话框里的显式入口尚未 GA。
01
Mode one · 运行面协作

同一个工作流，开始跨 UI、终端和 CI 使用。

difyctl 让工作流可以被开发脚本、个人 Agent 与流水线直接调用；scoped env 与统一错误让自动化边界更清晰。

Builder

在 UI 中设计与调试，保留可视化上下文。

Operator

在 CLI 中运行、观察、组合和批处理。

Pipeline

在 CI 中触发、回归、审计并形成发布门禁。

推演依据：Dify 1.15.0 的 difyctl、scoped environment variables、CLI/HTTP 统一错误与可见 reasoning。
新的协作不是把 UI 换成 CLI，而是让不同角色在不同执行面使用同一个工作流。设计者用 UI，运营与开发用 CLI，组织用 CI；工作流成为跨界面的共同资产。

HITL 的价值不是“Agent 不行就叫人”。真正成熟的设计是 Agent 先把材料准备到可判断状态，表单承载结构化交付，人做高价值判断，Workflow 带着决定恢复执行，并把证据留下。

Mode three · 可复用数字同事

Agent 开始有稳定的运行环境、方法和名册。

ENV

Linux Sandbox

代码与 shell 在隔离环境执行，能力边界可管理。

METHOD

Skills

把 SOP、工具、脚本与知识打包为可分发能力。

IDENTITY

Agent Roster

跨工作区管理、复用与引用 Agent。

COACH

Builder Agent

通过对话配置环境、安装包、创建 Skills 与文件。

组织可以管理的不再只有 Prompt，还包括运行环境与能力包。

推演依据：Dify 1.16.0 Dify Agent (Beta) 的 sandbox、Skills、Agent roster 与 builder agent。
角色定义不再只是一个 prompt。一个可复用 Agent 同时包含基础提示、Skills、文件、工具、知识和运行环境。这让“谁来做”第一次成为可以被组织管理、复用和审计的数字资产。
Mode four · 组织协议

Workflow 开始承担跨角色交接与留痕。

HUMAN

目标与授权

定义结果、边界、预算与需要人工判断的节点。

AGENT

自主执行

在沙箱里调用 Skills、文件、工具与知识。

WORKFLOW

控制面

编排顺序、变量、重试、HITL、归档与可观测。

MCP

组织接口

结构化输出、版本协商与动态身份传递。

推演依据：Dify 1.16.0 的 Agent Workflow integration、workflow-as-MCP structured output、dynamic headers 与 run archives。
Workflow 的价值正在从“自动化一条流程”上升为“定义谁在何时、带什么身份、交付什么结构、留下什么证据”。MCP 让这份协议可被外部系统调用，Agent 节点让自主执行成为协议的一部分。
Target operating model · 目标协作模型

把生产系统拆开看，至少有四类责任。

01 / HUMAN

意图与问责

设定目标、批准高风险动作、处理例外，并对业务结果负责。

02 / AGENT

探索与执行

在受控环境中分解任务、调用 Skills、产出结构化交付物。

03 / WORKFLOW

编排与证据

管理状态、顺序、HITL、重试、归档、成本与可观测性。

04 / GOVERNANCE

权限与边界

插件分级、策略即代码、身份、数据流、签名、监测与处置程序。

权限、状态和责任分别落到位，协作系统才容易运营。

这页把两部分合起来。人的职责是意图与问责；Agent 负责探索执行；Workflow 负责状态和证据；治理负责权限与边界。四层不是组织架构，而是每个生产系统必须清楚回答的责任问题。
