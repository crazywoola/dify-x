TRUST / COMPOSABILITY
今天不逐个罗列安全工具，只回答两个实际问题。第一，开放插件进入企业之前，哪些规矩必须先立起来；第二，Dify 1.15 和 1.16 带来的新能力，会怎样重新安排人与 Agent 的分工。
01 · 先定义问题

企业装上一个插件，等于交出了三把钥匙。

{ }

代码运行权

允许它在运行时执行依赖、命令、二进制和模型调用。

↕

数据处理权

允许它读取文件、对话、知识库、凭证和工作区信息。

◎

外部连接权

允许它访问第三方 API、任意 URL、远端服务和下游处理者。

“不以规矩，不能成方圆。”开放生态也要先把权限、证据和责任说清。

典故：《孟子·离娄上》。
插件看起来只是一个功能入口，装进企业后却拿到了三把钥匙：可以运行代码，可以处理数据，也可以连到外部服务。《孟子》说“不以规矩，不能成方圆”，开放生态也是这个道理。后面的治理办法，都从这三类权限出发。
Observed risk · 已发现的风险

最先暴露的不是高明攻击，而是整个开发目录被塞进安装包。

抽样中已发现开发残留和敏感材料；daemon（守护进程）解包后，它们就会落到用户机器上。

.env / 密钥 .git/ / 历史 .venv/ / 编译产物 *.exe / *.so 日志 / 缓存 / IDE
PACKAGE → DAEMON → TENANT
来源：飞书《Dify 插件生态治理提案》执行摘要与“风险敞口”章节；此页列举文档中已经发现的包污染类型。
抽样包里已经出现 .env、.git、虚拟环境、日志和平台二进制。这些文件会跟着安装包一起进入用户机器。问题首先出在发布流程没有把好关，不能只归咎于作者粗心。

Six lenses · 六种视角

一个插件摆到桌上，六个团队各有一本账。

01 / SECURITY

供应链安全

密钥、二进制、依赖漏洞、运行时行为与上架后变质。

02 / LEGAL

法务与合规

平台责任、子处理者、知识产权、下架与申诉程序。

03 / PRODUCT

产品信任

徽章、能力清单、数据流、签名、SBOM 与风险提示。

04 / ENGINEERING

工程效率

本地预检、分级 CI、可定位报告、构建产物（artifact）复用与自动标签。

05 / COMMUNITY

社区可持续

低风险尽量少打断、规则可提前试跑、例外有章可循。

06 / BUSINESS

商业准入

让企业安全团队能够评估、签字，并且定期复核。

安全团队查供应链，法务团队理清责任，产品团队要把证据摆给用户看。工程和社区关心规则能不能长期执行，商业团队则要回答客户最直接的问题：这个插件能不能通过安全评估。六本账要合在一起算，不能各管一摊。
Trust gap · 信任落差

话说得多满，证据就要做得多实。

对外承诺（Marketing）
“Safer, more reliable”

用户听到的是发布者（publisher）作出的安全承诺。

实际工作（Operations）
解包 + 清单校验

眼下更像托管服务商（hosting service），还缺安全扫描和持续监测。

主站已有 SOC 2、ISO 27001 与 GDPR 体系，插件市场不能成为尽调空白区。

来源：飞书提案 §2.4 与 §3.1。法律定位需由适用法域下的专业法务最终判断。
主站已经有 SOC 2、ISO 27001 和 GDPR 体系，客户自然会用同样的标准来看插件市场。平台究竟更像托管服务商还是发布者，不只看条款，也看审核有多深、怎么推荐、挂什么徽章、对外怎样承诺。过渡期文案要不要收一收，应当由法务和产品一起拍板。
Responsibility chain · 责任链

“名不正，则言不顺”：谁决定、谁处理、谁转交，必须写清。

01 · CUSTOMER

业务用户 / 控制者（Controller）

决定处理目的，承担内部授权与业务结果责任。

02 · PLATFORM

Dify / 处理者（Processor）?

负责执行、凭证、分发与审计；角色随部署方式变化。

03 · AUTHOR

插件作者 / 子处理者（Sub-processor）?

决定具体实现与下游调用；协议和告知仍需补强。

04 · DOWNSTREAM

第三方 API / 再处理者

数据流向、区域、保留策略与再委托需要透明。

隐私（Privacy）

类型、目的、第三方。

知识产权与许可

权属、许可、分发权。

正当程序

通知、申辩、复议。

证据（Evidence）

签名、SBOM、时间戳。

来源：飞书提案 §3.2–§3.5；典故：《论语·子路》。图中问号表示角色需按具体处理场景与合同关系认定。
孔子说“名不正，则言不顺”。放到数据治理里，就是先把角色叫对、责任分清。图中的问号不能轻易去掉，因为 GDPR 角色要结合部署方式和合同关系判断。法务至少要看清四件事：谁定目的，谁真正处理，数据又交给了谁，出了问题如何通知、申诉和举证。
Risk proportionality · 风险分级

“权，然后知轻重；度，然后知长短。”插件审核也该先分风险。

档位典型行为机器检查人工审核用户感知
L1固定 HTTPS API；无本地副作用硬拒绝 + Semgrep 警告抽样 20%基础验证 · 不额外打断
L2用户文件 / URL；上传下载与富文本解析依赖 Medium+ 阻断；警告需说明全量简审标准风险 · 数据去向可见
L3命令、SQL、SSH、浏览器、本地文件、任意 HTTP警告升级为阻断；Low+ 依赖阈值深度审核 + 动态测试高风险 · 加强审查 + 安装二次确认

原则：低风险不增摩擦，高风险不靠运气。

来源：飞书提案附录 C《风险分级标准》及其嵌入表格；典故：《孟子·梁惠王上》。
《孟子》说，称过才知道轻重，量过才知道长短。插件治理也一样，不能不分青红皂白地一律从严。作者自评和机器检查取较高档，审核人还可以调档。这样既守住安全，也不让普通贡献者平白承担 L3 的成本。
Lifecycle governance · 全生命周期

“慎终如始，则无败事。”一次 CI 通过，不能换来永久可信。

01

作者

严格模板、.difyignore、风险自评、能力与隐私声明。

02

构建

源码 URL、提交 SHA（commit SHA）、锁定依赖与包指纹。

03

入库 CI

密钥、二进制、依赖、SAST、插件清单（manifest）与风险档位。

04

上架

官方 / 合作伙伴（Partner）/ 社区分层，展示徽章与可信信息。

05

安装运行

L3 二次确认；最小能力；核对沙箱与出站行为。

06

持续监测

新 CVE、异常调用、降级、SLA、通知与复议。

典故：《道德经》第六十四章。
《道德经》说“慎终如始，则无败事”。CI 通过，只能证明检查当时没有发现问题。版本会变，依赖会冒出新漏洞，作者凭证也可能被盗。治理必须从作者提交一直管到安装、运行和后续更新，上架后的监测与处置，和入库扫描同样重要。

作者本地先跑

同一套审核工具（review-plugin）在作者机器与 CI 运行，少走弯路。

证据可验证

报告包含 SHA-256、工具版本、时间戳与命中位置。

来源：飞书提案附录 D《入库门禁技术设计》；典故：《论语·卫灵公》。
“工欲善其事，必先利其器。”这九个阶段不是为了罗列工具，而是要办成三件事：先按风险档位决定检查力度；作者本地和 CI 使用同一套规则；每次结论都带上指纹、版本和命中位置。审核人看到证据，才知道这份结论为什么可信。

Governance stack · 三层治理

机器守底线，人看具体情况，程序给争议一个说法。

LAYER 01

机器门禁

硬拒绝、分级阈值、签名、SBOM、依赖与静态扫描。

目标：任何插件都不能越过底线。

LAYER 02

人工判断

L3 深度审核、动态测试、安全例外与双人签字确认（sign-off）。

目标：处理机器看不懂的具体情况和真实意图。

LAYER 03

法律程序

作者行为准则（Author CoC）、协议、通知、申辩、决定、复议与安全例外。

目标：让每次处置都有依据、能够执行、经得起复核。

只靠机器，容易误报，也容易把规则做死；只靠人工，标准会忽高忽低，任务还会越积越多。更稳妥的办法是各司其职：机器守住硬底线，人判断具体情况，正式程序负责通知、申辩和复议。这样既能处置问题，也能维持与作者的长期关系。

Trade-offs · 治理也要被治理

“一张一弛，文武之道也。”规则有力度，也要给迁移留余地。

FRICTION

门禁过严抑制贡献

L1 尽量少打断；官方仓试运行 2 周再推广；社区仓前 30 天只警告、不阻断。

FALSE POSITIVE

误报与例外

每次命中都给说明和文档；支持内联豁免；安全例外需至少 2 名维护者（maintainer）说明业务理由。

MIGRATION

现有作者迁移成本

新规则先管新提交；存量先扫描并展示结果；严重（Critical）问题给 90 天修复期。

DUE PROCESS

封禁与下架争议

通知 → 申辩 → 决定 → 复议；安全例外可先下后告，但须事后说明。

CI 可能从约 3 分钟增至 6–8 分钟；通过并行检查和复用构建产物控制成本。

来源：飞书提案 §5《实施风险与权衡》嵌入表格；典故：《礼记·杂记下》。
《礼记》讲“一张一弛”，规则也是如此。官方仓先试两周，社区仓前三十天只告警；误报要有说明、有文档、能申请豁免；安全例外至少两名维护者批准；存量严重问题给九十天整改。要求越严，越要提前告诉作者怎样通过、怎样申诉。

需要法务 × 产品共同决定
共同负责人工程安全法务产品 / 设计开发者关系
来源：飞书提案 §6《需要决策事项》；建议项与待决项按研究稿口径呈现。
扫描器和模板，项目组可以继续做。但这三项选择会改变风险承诺、产品展示和运营成本，必须得到管理层明确授权。尤其是 L3 是否默认人工预审、过渡期营销文案是否调整，不能让审核人到现场临时决定。

Feature spotlight · AI 自动生成应用

一句话先变成计划，再变成能够继续修改的 Workflow / Chatflow。

01 · UNDERSTAND

理解需求

结合工作区信息提出可做的方案，并判断应用模式。

02 · PLAN FIRST

先展示节点计划

生成画布前先看真正要建的节点；方向不对，可以马上叫停。

03 · BUILD + CHECK

边生成，边检查

配置节点并验证工具和连接；出错时点明受影响的节点。

04 · APPLY / REFINE

应用后继续修改

/refine 列出增删改，生成结果仍然可以编辑。

从“逐个画节点”转向定目标、审计划、批权限、管例外生成结果照常接受治理
来源：Dify 1.16.0 Release「Smarter AI Workflow Generation」与 PR #38175；当前入口为 ⌘K → /create、/refine，Create-App 对话框入口尚未 GA。
这里的“自动生成应用”指 Workflow 和 Chatflow。用户在 Command K 中输入 /create，用一句话说清需求；系统先判断模式，列出真正会生成的节点，再去搭建和验证画布。/refine 会说明改了什么。变化不在应用归谁，而在第一稿由谁来写：机器搭流程，人定目标、审计划、批工具、管例外。目前这项功能仍在预览阶段，Create-App 对话框里的明确入口还没有正式开放。
01
Mode one · 运行面协作

同一个工作流，可以在界面、终端和流水线上各展所长。

difyctl 让开发脚本、个人 Agent 和流水线都能直接调用工作流；作用域环境变量（scoped environment variables）和统一错误让自动化边界更清楚。

设计者（Builder）

在 UI 中设计和调试，保留可视化信息。

操作者（Operator）

在 CLI 中运行、观察、组合和批量处理。

流水线（Pipeline）

在 CI 中触发、回归、审计，并把住发布关口。

判断依据：Dify 1.15.0 的 difyctl、作用域环境变量、CLI/HTTP 统一错误与可见推理过程。
新的协作不是用 CLI 取代 UI，而是同一个工作流可以在不同地方发挥作用。设计者在 UI 里搭建，运营和开发在 CLI 里运行，组织再用 CI 做回归和审计。大家使用的是同一份工作流，不必各自重做一套。
02
Mode two · 结构化人机协作

人工参与（Human-in-the-Loop）应当是正式交接，不只是失败后的补救。

AGENT

准备材料

收拢上下文，给出候选答案和风险说明。

→
HITL FORM

表单交件

用选项、文件和明确字段把材料交到人手中。

→
EXPERT

定夺 / 补充

在关键时点作判断，不必从头重做。

→
WORKFLOW

继续执行

带着人的决定往下走，并把过程记下来。

判断依据：Dify 1.15.0 更丰富的 Human-in-the-Loop 表单与推理过程留存能力。
HITL 不是“Agent 干不动了，再叫人来救火”。更好的分工是：Agent 先把材料备齐，表单把选项和文件送到专家面前，人只在关键处定夺，工作流带着决定继续运行，同时把过程留档。人该拍板时拍板，机器该干活时干活。
03
Mode three · 可复用数字同事

Agent 有了固定环境、做事方法和名册，才真正可以复用。

ENV

Linux Sandbox

代码与 shell 在隔离环境中执行，能做什么可以管住。

METHOD

Skills

把 SOP、工具、脚本与知识打包为可分发能力。

IDENTITY

Agent Roster

跨工作区登记、管理、复用和引用 Agent。

COACH

Builder Agent

通过对话配置环境、安装包、创建 Skills 与文件。

组织要管的不再只有提示词（Prompt），还要管运行环境和能力包。

判断依据：Dify 1.16.0 Dify Agent (Beta) 的 sandbox、Skills、Agent roster 与 builder agent。
一个角色不再只靠一段提示词来定义。可复用 Agent 还包括 Skills、文件、工具、知识和运行环境。这样一来，“谁来做”不再是口头安排，而是组织可以登记、复用和审计的一套明确配置。
Mode four · 组织协议

工作流开始把跨角色的交接、状态和证据串起来。

HUMAN

目标与授权

定义结果、边界、预算与需要人工判断的节点。

AGENT

自主执行

在沙箱里调用 Skills、文件、工具与知识。

WORKFLOW

流程控制

安排顺序、变量、重试、HITL、归档和运行观测。

MCP

组织接口

结构化输出、版本协商与动态身份传递。

判断依据：Dify 1.16.0 的 Agent Workflow integration、workflow-as-MCP structured output、dynamic headers 与 run archives。
工作流不只是把步骤自动跑一遍。它还要说明：谁在什么时候出手，用什么身份，交付什么格式，留下哪些证据。MCP 让外部系统可以照着这套约定调用，Agent 节点则把自主执行纳入同一套流程。
Target operating model · 目标协作模型

系统要跑得稳，四类责任缺一不可。

01 / HUMAN

意图与问责

设定目标、批准高风险动作、处理例外，并对业务结果负责。

02 / AGENT

探索与执行

在受控环境中拆解任务、调用 Skills，交出结构化结果。

03 / WORKFLOW

编排与证据

管理状态、顺序、HITL、重试、归档、成本和运行观测。

04 / GOVERNANCE

权限与边界

插件分级、策略即代码、身份、数据流、签名、监测与处置程序。

谁给权限、谁管状态、谁负责任，都有明确去处，系统才管得住。

把前后两部分合起来看，人的职责是定目标、作决定、负责任；Agent 负责探索和执行；工作流负责状态和证据；治理负责权限和边界。这四层不是组织架构图，而是每个生产系统都必须答清楚的四类问题。
Action plan · 30 / 90 / 365

未来一年：先把规矩立住，再逐步扩大协作。

NEXT 30 DAYS

跑通最小治理流程

  • 清点插件与权限范围
  • 定义 L1/L2/L3
  • 本地预检 + 审核清单
  • 标出必须 HITL 的动作
交付：插件清单、风险档位、负责人、例外流程
NEXT 90 DAYS

把规则写进协作流程

  • 分级 CI 与证据报告
  • Agent 名册 + Skills 目录
  • UI / CLI / CI 共用一套工作流
  • HITL 表单与运行归档
交付：可复用 Agent、可追溯流程、可验证插件
NEXT 365 DAYS

把信任做成用户可用的功能

  • 签名、SBOM、持续 CVE
  • 插件市场可信信息中心
  • MCP 组织接口与动态身份
  • 透明指标与响应 SLA
结果：参与者更多，责任边界依然清楚
最后给管理者一条可以按月推进的路径。三十天先做清点和分级，九十天内把规则写进 CI、Agent 名册、Skills 和 HITL；一年之内再补签名、SBOM、MCP 身份和可信信息中心。顺序很重要：先立规矩，再逐步放开协作。
